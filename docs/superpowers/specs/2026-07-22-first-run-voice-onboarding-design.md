# 首次使用语气建档引导设计

## 目标

用户安装 `hongduixiang` 后，第一次调用 `girlfriend-reply-coach` 或 `create-hong` 时，如果尚无有效的本人语气档案，系统主动引导用户通过经授权的聊天记录建立档案。用户可以跳过，并继续获得明确标注的中性口语回复。

安装动作本身不启动交互；首次引导在 Skill 第一次被调用时触发。

## 有效档案判定

默认私有数据根目录为 `~/.codex/private/girlfriend-reply-coach`，也可由 `GIRLFRIEND_REPLY_COACH_DATA` 指定。

本人语气档案只有在 `self-voice.json` 存在、结构可读且至少包含一项有效 `patterns` 时才算已完成初始化。只有空数组的初始化文件属于空壳，仍需显示首次引导。损坏或结构不合法的文件不被静默接受，应提示修复或重新建档。

## 首次交互

检测到档案尚未完成时，显示：

```text
还没有你的语气档案。建立后，我可以根据聊天记录学习你的称呼、句长、语气词、标点和表情习惯。

请选择：
1. 导入 TXT / JSON / JSONL 聊天记录
2. 直接粘贴几段聊天
3. 暂时跳过，使用自然的中性口语
```

### 导入文件

询问聊天文件路径和用户是否有权使用资料。只有明确授权后才读取。解析后展示发送者及各自文本消息数，让用户明确确认本人映射；不得根据聊天内容猜测身份。系统消息和媒体不计入语气证据。

### 直接粘贴

先询问授权，再接收聊天片段。采用与文件导入相同的发送者确认、证据过滤和语气提取规则。

### 暂时跳过

继续当前回复任务，使用自然、简短、无专属称呼的中性口语，并明确说明本次未使用个人语气档案。跳过不创建伪造模式，也不把空壳档案标记为初始化完成。下次新的 Skill 调用仍可再次显示引导。

## 与对象专属 Skill 的关系

本人语气初始化只是创建 `hong-<slug>` 的一部分。用户仍需通过 `create-hong` 完成：

1. 当前对象代号和 slug；
2. 聊天资料授权；
3. 本人与对象发送者映射；
4. 双方至少各一条可识别文本消息；
5. `profile.json` 和 `voice-delta.json`；
6. 摘要预览与明确确认；
7. 生成并安装绑定档案版本的 `hong-<slug>`。

不得因为本人语气档案已存在而跳过对象档案门槛。

## 重新学习

用户说“重新学习我的语气”“更新我的语气档案”或表达同等意图时，重新进入导入流程。新证据不静默覆盖旧模式；冲突需要保留来源、降低置信度或等待确认。依赖本人语气档案的已生成对象 Skill 应标记为需要重新生成。

## 组件变化

- 在本地档案模块增加确定性的本人语气档案状态检查，区分缺失、空壳、有效和损坏。
- 在 `girlfriend-reply-coach` 与 `create-hong` 中加入首次调用检查及三选一引导。
- 在 README 中说明安装后首次调用体验、跳过效果和重新学习入口。
- 使用契约测试锁定引导文案和行为；使用单元测试验证四种档案状态。

## 错误处理

- 无权限：不读取资料，允许返回选择界面或中性模式。
- 文件不存在或格式不支持：说明具体问题并允许重新提供，不生成档案模式。
- 无法确认发送者：暂停提取，要求用户确认映射。
- 没有可用本人文本：档案保持未初始化。
- 档案损坏：报告损坏状态，不将其当成空白新档案覆盖。

## 隐私与安全

- 所有语气档案默认仅保存在 Git 仓库外的本地私有目录。
- 未经授权不读取聊天，不把聊天正文写入公开 Skill。
- 只学习用户自己的表达习惯，不模拟对象人格。
- 不学习或复现辱骂、操控、威胁、跟踪、欺骗、消息轰炸等有害习惯。

## 验收标准

1. 缺失或空壳 `self-voice.json` 会触发首次引导。
2. 含有效模式的档案不会重复触发引导。
3. 损坏档案会收到明确错误，不会被覆盖。
4. 用户可选择文件、粘贴或暂时跳过。
5. 跳过后当前任务仍能继续，但输出明确标注未使用个人语气档案。
6. 重新学习会使依赖旧档案的对象 Skill 进入待重新生成状态。
7. 对象专属 Skill 的原有建档与确认门槛保持不变。
