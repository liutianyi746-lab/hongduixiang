# `create-hong` 建档门槛与专属 Skill 生成设计

## 目标

将对象专属回复从“运行时临时切换”改为“先建档、再生成、后调用”。只有完成资料授权、双方映射、本人语气档案、对象档案、对象专属语气差异和用户预览确认后，系统才生成真实、独立、可调用的 `hong-<slug>` Skill。

生成的 Skill 是只保存绑定信息和运行规则的薄 Skill。真实聊天、人物档案和语气数据始终位于 Git 仓库之外的私有目录。

## 组件职责

### `create-hong`

专属 Skill 生成器，负责：

- 接收对象代号与 slug；
- 确认资料使用授权；
- 导入或接收聊天资料；
- 确认双方发送者映射；
- 建立或更新本人通用语气、对象事实与对象专属语气差异；
- 展示生成摘要并等待用户确认；
- 校验生成门槛；
- 将 `hong-<slug>` 安装到 Codex、Claude Code 或两者。

### `hong-<slug>`

固定绑定一个对象，只负责运行时联动：

1. 校验私有档案存在且版本匹配；
2. 使用 `goutoujunshi` 生成关系判断和原始话术；
3. 建立语义锁；
4. 读取本人通用语气和当前对象专属语气差异；
5. 忠实改写并输出一条可发送回复。

它不得建档、切换对象、猜测对象、内嵌私人材料或在档案不可用时生成普通临时回复。

### `girlfriend-reply-coach`

保留为底层语气适配规则和数据工具。它不再把未建档对象当作可运行的个性化对象。`create-hong` 生成的子 Skill 使用其语义保真和隐私约束。

## 创建入口

用户可使用：

```text
/create-hong xiaoyu
```

Codex 中也可自然语言调用 `create-hong`。slug 必须由小写字母、数字和单连字符组成，首尾为字母或数字；保留字 `hong`、`create-hong`、`update-hong` 不可使用。生成名称固定为 `hong-<slug>`。

## 创建流程

### 1. 基础信息

只要求两个必填项：

- 对象匿名代号；
- 对象 slug。

不得要求真实姓名。其他背景可以缺省，不用 MBTI、星座或主观评分作为生成门槛。

### 2. 授权与材料

用户必须明确确认有权使用资料。支持：

- WeFlow TXT、JSON、JSONL；
- QQ、微信等可被现有导入器解析的文本；
- 直接粘贴的聊天片段；
- 用户明确输入的语气纠正。

生成首版专属 Skill 必须至少有双方各一条可识别的文本消息。媒体、系统消息和撤回提示不计入该门槛。聊天记录不足时可以继续补充，但不能生成空壳 Skill。

### 3. 双方映射

展示检测到的发送者与文本消息数，让用户明确选择：

- 哪些发送者别名属于本人；
- 哪些发送者别名属于当前对象。

双方集合必须非空且不重叠。未知发送者不得通过内容自动猜测身份。

### 4. 建立档案库

私有目录结构：

```text
~/.codex/private/girlfriend-reply-coach/
├── self-voice.json
└── people/<slug>/
    ├── profile.json
    ├── voice-delta.json
    └── generation.json
```

`self-voice.json` 保存跨对象稳定的本人表达习惯。`profile.json` 保存对象代号、发送者映射、授权状态、来源摘要和必要事实。`voice-delta.json` 保存用户面对该对象时相对于通用语气的表达差异。

每项语气模式必须包含：内容、证据来源、置信度、适用情境和更新时间。系统不凭单条消息宣称高置信度模式；证据不足的特征标记为待验证。

### 5. 预览确认

生成前展示：

- 对象代号和 slug；
- 双方别名映射与文本消息数；
- 本人通用语气摘要；
- 当前对象专属语气摘要；
- 缺失或冲突信息；
- 将安装的 Skill 名称与目标目录。

用户必须明确确认“生成”。用户纠正后先更新档案，再重新展示摘要，不能沿用纠正前的确认。

### 6. 生成门槛

以下条件全部满足才允许生成：

1. slug 合法且未使用保留字；
2. 授权状态为已确认；
3. 本人和对象的发送者映射均非空且不重叠；
4. 双方各至少存在一条纳入证据的文本消息；
5. `self-voice.json` 存在且通过结构校验；
6. `profile.json` 存在且通过结构校验；
7. `voice-delta.json` 存在且通过结构校验；
8. 用户确认的摘要哈希与当前三个档案一致；
9. 安装目标明确为 `codex`、`claude` 或 `both`。

失败时一次列出所有未满足条件，不写入任何 Skill 目录。

## `generation.json`

成功生成后写入：

```json
{
  "schema_version": 1,
  "slug": "xiaoyu",
  "skill_name": "hong-xiaoyu",
  "profile_version": "档案组合哈希",
  "confirmed_summary_hash": "用户确认摘要哈希",
  "generated_at": "ISO-8601",
  "install_targets": ["codex", "claude"],
  "skill_paths": ["绝对安装路径"],
  "needs_regeneration": false
}
```

档案发生任何有效更新时，将 `needs_regeneration` 设为 `true`。重新生成成功后写入新版本并恢复为 `false`。

## 生成的薄 Skill

每个安装目录只包含：

```text
hong-<slug>/
├── SKILL.md
└── binding.json
```

`binding.json` 只保存：schema 版本、slug、期望档案版本和私有数据根目录的解析规则；不保存真实姓名、聊天、人物事实、语气摘要或绝对私有路径。

`SKILL.md` 的 frontmatter 名称固定为 `hong-<slug>`，描述只使用匿名代号。正文必须规定：固定对象、先校验绑定、必须联动 `goutoujunshi`、只读取对应档案、语义锁优先、不得跨人物回退。

## 安装与事务

默认安装目标为用户在创建流程中选择的 `codex`、`claude` 或 `both`：

- Codex：`~/.codex/skills/hong-<slug>`
- Claude Code：`~/.claude/skills/hong-<slug>`

已存在时默认拒绝覆盖。用户明确确认更新后才允许事务式替换；`both` 任一目标失败时恢复所有旧版本。

## 调用规则

成功生成后：

```text
/hong-xiaoyu <当前聊天或事件>
```

运行前必须检查：

- `binding.json` 合法；
- 私有三份档案存在；
- `generation.json` 存在；
- `needs_regeneration` 为 `false`；
- 当前档案组合哈希等于绑定的期望版本。

任一条件不满足就停止个性化回复，提示运行 `/create-hong <slug>` 或 `/update-hong <slug>`。不能输出临时回复，不能切换到其他对象，也不能使用通用档案冒充专属档案。

如果 `/hong-xiaoyu` 本身尚未安装，宿主通常不会加载该命令；文档和生成器应提示用户先运行 `/create-hong xiaoyu`，而不能宣称已切换对象。

## 更新

入口：

```text
/update-hong xiaoyu
```

更新流程复用导入、映射、档案分析和预览确认步骤。任何档案变化立即使旧 Skill 失配；确认后重新生成并事务式替换指定安装目标。

用户说“不像我”时，系统先区分通用语气纠正和对象专属语气纠正，再更新相应档案。不得直接编辑薄 Skill 来掩盖档案问题。

## 删除与列出

- `/list-hong`：只列出 generation 记录有效的匿名代号、slug、安装目标和是否需要重新生成；不展示私人摘要。
- `/delete-hong <slug>`：先展示将删除的私有人物目录、generation 记录和两个可能的 Skill 安装目录，获得明确确认后删除；不得影响其他对象或 `self-voice.json`。

## 隐私与安全

- 所有原始材料和派生档案默认只在本地处理；未经授权不发送第三方。
- 公开仓库、生成 Skill 和 Git 历史不得包含真实人物资料。
- 生成的 Skill 代表用户自己的表达适配器，不冒充对象，也不模拟对象人格。
- 不从照片外貌、MBTI、星座、回复速度或单句话推断人格、疾病、忠诚或爱意。
- 不复现辱骂、操控、威胁、跟踪、欺骗或连续轰炸等有害“语气习惯”。
- 涉及现实安全风险时保留 `goutoujunshi` 的安全提示，不得通过语气改写弱化。

## 验证

自动测试必须覆盖：

1. 未授权、未映射、缺任一档案或未确认摘要时拒绝生成且零写入；
2. 双方文本证据不足时拒绝生成；
3. 合法档案生成 `hong-<slug>/SKILL.md` 和 `binding.json`；
4. 生成物不包含聊天正文、真实别名、档案摘要或绝对私有路径；
5. `codex`、`claude`、`both` 安装路径正确；
6. 多人生成物固定读取各自 slug，不混用；
7. 档案更新后旧 Skill 检测版本失配并拒绝回复；
8. 事务式覆盖失败时恢复全部旧 Skill；
9. 删除只影响指定对象；
10. README 不再把 `/hong-<slug>` 描述为无需建档即可临时使用。

## 非目标

- 不训练或微调模型；
- 不把私人档案内嵌进生成 Skill；
- 不让一个 `hong-<slug>` 动态切换多人；
- 不在建档未完成时提供“先凑合用”的个性化回复；
- 不模拟、冒充或替代现实中的对象；
- 不自动联系任何真人。
